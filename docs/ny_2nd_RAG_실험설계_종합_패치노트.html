2. app/retrieval/backends.py — Backend 확장

현재 BGE-M3 전용 factory에 E5/SBERT/Jina/GTE backend를 추가합니다. 이미 DenseRetrievalBackend Protocol이 정의되어 있으므로, 이를 구현하는 class만 추가하면 pipeline의 나머지 코드는 변경할 필요가 없습니다.

from sentence_transformers import SentenceTransformer

class STModelRuntime:
    """SentenceTransformer 기반 공통 런타임 (E5/SBERT/Jina/GTE 용)"""
    def __init__(self, model_dir: Path):
        self.model = SentenceTransformer(str(model_dir), device="cuda")
        self.model_dir = model_dir

class STDenseBackend(DenseRetrievalBackend):
    """E5/Jina/GTE/Ko-SBERT 등 Dense 전용 backend"""
    def __init__(self, runtime: STModelRuntime):
        self.runtime = runtime
        self.model_dir = runtime.model_dir
    def encode_documents(self, texts):
        return self.runtime.model.encode(texts, normalize_embeddings=True)
    def encode_query(self, question):
        return self.runtime.model.encode([question], normalize_embeddings=True)[0].tolist()

# Factory 확장
def build_dense_backend(backend_name, model_dir, runtime_cache):
    if backend_name == "bgem3":
        return BGEM3DenseBackend(_get_bgem3_runtime(runtime_cache, model_dir))
    if backend_name in ("e5", "sbert", "jina_v3", "gte_multilingual"):
        cache_key = ("st", model_dir.resolve())
        if cache_key not in runtime_cache:
            runtime_cache[cache_key] = STModelRuntime(model_dir)
        return STDenseBackend(runtime_cache[cache_key])
    raise ValueError(f"Unsupported: {backend_name}")

pipeline.py 의 기존 코드는 build_dense_backend()를 호출하기만 하므로, 여기에만 새 backend를 등록하면 자동으로 동작합니다.

def build_reranker_backend(backend_name, corpus_store, model_dir) -> RerankerBackend:
    if backend_name == "bge_reranker":
        return BGERerankerV2M3(corpus_store, model_dir)
    if backend_name == "bge_reranker_v2.5_gemma2":
        # 동일 BGERerankerV2M3 클래스 사용, 모델 dir만 다름
        return BGERerankerV2M3(corpus_store, model_dir)
    if backend_name == "jina_reranker_v3":
        return JinaRerankerV3(corpus_store, model_dir)
    if backend_name == "minilm_l6":
        return MiniLMReranker(corpus_store, model_dir)
    raise ValueError(f"Unsupported: {backend_name}")

        self.model_dir = Path(model_dir)
        # transformers AutoModelForCausalLM + EXAONE chat template
        ...

def build_llm_backend() -> LLMBackend:
    backend = os.getenv("FAIRDATA_LLM_BACKEND", "qwen")
    model_dir = resolve_llm_model_dir()
    if backend == "qwen": return QwenBackend(model_dir)
    if backend == "exaone": return ExaoneBackend(model_dir)

6) Ver2 - Retrieval 방식별 적용 범위와 실험 후보군

6-1) Dense / Sparse / Multi-Vector 구분 원칙

ver2에서는 "임베딩 모델을 바꾼다"는 말을 검색 방식별로 분해해서 봐야 합니다. 현재 ny_2nd 코드 기준으로는 Dense Search, Sparse Search, Multi-Vector Search가 요구하는 모델 조건이 서로 다릅니다.

검색 방식일반 임베딩 모델 사용 가능 여부대표 예시현재 ny_2nd 코드 기준 해석
Dense Search가능OpenAI embedding, bge-small, bge-base, e5, gte, jina, ko-sbert일반 임베딩 벡터만 만들 수 있으면 대부분 붙일 수 있음
Sparse Search조건부 가능BM25, SPLADE, BGE-M3 sparseBM25는 모델 없이 가능하지만, learned sparse는 전용 모델/출력이 필요함
Multi-Vector Search일반적으로 어려움ColBERT, BGE-M3 multi-vector문서당 여러 벡터를 내는 전용 출력 구조가 필요함

즉 dense-only 모델 = retrieval 전체 대체가 아닙니다. 대부분의 일반 임베딩 모델은 dense 경로만 직접 대체할 수 있고, sparse와 multi-vector는 별도 전략이 필요합니다.

6-2) 현재 코드 기준으로 왜 모든 조합을 바로 못 돌리는가

  • Dense Search는 현재 구조에서도 비교적 쉽게 교체 가능합니다. SentenceTransformer 계열 또는 유사 인터페이스만 맞추면 됩니다.
  • Sparse Search는 현재 코드에서 사실상 BGEM3HybridModel 구현에 묶여 있습니다. 따라서 e5 + sparse, gte + sparse 같은 조합은 현재 구조상 성립하지 않습니다.
  • Multi-Vector Search도 현재 구현은 BGE-M3 계열 전용입니다. 일반 dense 모델은 문서당 단일 벡터만 내므로 바로 대체할 수 없습니다.
  • 따라서 "모든 모델을 동일한 hybrid 설정으로 공정 비교"는 현재 코드 기준으로 불가능하고, dense-only 비교군과 full-hybrid 비교군을 분리해야 합니다.

2. app/retrieval/backends.py - BM25 sparse backend 추가

  • 현재 sparse backend는 BGE-M3 전용입니다.
  • BM25SparseBackend를 추가하면 e5 + BM25, jina + BM25, gte + BM25 같은 dense+lexical hybrid 실험이 가능합니다.
  • 가장 단순한 방식은 rank_bm25 또는 유사 라이브러리 기반으로 문서 토큰 인덱스를 유지하는 것입니다.
def build_sparse_backend(backend_name, model_dir, runtime_cache):
    alias = _resolve_dense_alias(backend_name)
    if alias in {"bgem3", "bge_m3", "upskyy_bgem3_ko"}:
        return BGEM3SparseBackend(_get_bgem3_runtime(runtime_cache, model_dir))
    if alias == "bm25":
        return BM25SparseBackend(_get_bm25_runtime(runtime_cache))
    raise ValueError(f"Unsupported sparse backend: {backend_name}")

7) 결론

  • ny_2nd (203cd7b)는 Interface Protocol + Factory 패턴이 이미 도입되어 있어 축별 실험 설계에 매우 유리한 구조입니다.
  • 임베딩: config.py의 상수를 env로 전환하고 backends.py에 STModelRuntime을 추가하면 8종 모델 비교가 가능합니다.
  • Reranker: backends.py factory에 v2.5-gemma2/jina-reranker-v3을 등록하면 7종 비교가 가능합니다.
  • LLM: generator.py에 LLMBackend Protocol 도입 후 Qwen3-8B/EXAONE-3.5-7.8B 등을 실험할 수 있습니다.
  • 총 수정 대상: config.py (env 전환), backends.py × 2 (factory 확장), reranker.py (새 클래스), generator.py (추상화) — 5~6개 파일.
  • Ver2 관점: 일반 dense 모델은 dense-only 또는 dense+BM25로 비교하고, learned sparse/multi-vector는 BGE-M3 계열과 분리해서 다뤄야 공정한 해석이 가능합니다.
  • 최종 제출안은 성능뿐 아니라 지연시간, 근거충실도, 재현성 로그까지 함께 고정해야 안정적입니다.
