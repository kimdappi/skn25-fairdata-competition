# QA 데이터셋 생성 가이드라인

## 목적
RAG 파이프라인의 **Retrieval(Recall@5, MRR)** 성능 및 **Generation(BERTScore, F1)** 품질을 종합적으로 평가하기 위한 통합 데이터셋 구축 방법론

---

## 1. QA 쌍의 구조

```json
{
  "id": "Q-S-001 / Q-M-001",
  "query": "사람이 자연어로 물어볼 법한 질문",
  "source_doc": "의결서 제목 (복수 의결서의 경우 여러 개 나열)",
  "answer_chunks": [
    {
      "chunk_id": "chunk_id_1",
      "answer_reason": "각 chunk가 정답인 이유 (기업명과 위반사실 포함)"
    },
    ...
  ],
  "reference_context": "검색 모델이 참조해야 할 **5개 내외**의 정답 청크 원문의 결합 (Generation 평가용)",
  "reference_answer": "Gemini 3.1 등을 활용해 요약한 고품질 모범 답안 (Generation 평가 골드 스탠다드)"
}
```

## 2. 질문(query) 작성 원칙

### 핵심 원칙: "사람이 실제로 입력할 법한 자연어 질의"

1. **기계적 파편 및 기호 사용 엄격 금지**
   - **말줄임표(`...`) 금지**: 문장 중간이나 끝에 생략 부호를 절대 사용하지 않음. 문장을 완결된 형태로 작성.
   - **번호 및 기호(`- 1.`, `가.`) 금지**: 문서의 번호 체계가 질문에 포함되지 않도록 함.
   - **특정 섹션 지칭 금지**: "의결서 주문에 명시된", "위법성 판단 섹션에서"와 같이 문서의 특정 위치를 지칭하는 표현 배제. 사용자는 문서 구조를 모른다고 가정함.

2. **구체적 인용 지양 및 자연스러운 재구성**
   - 문구를 그대로 따옴표(`''`)로 가져와서 "~내용을 포함한"이라고 묻는 대신, 해당 내용의 **의미(핵심 위반 행위)**를 파악하여 질문에 녹여냄.
   - (예시) "'- 1. 피심인은 ...' 시정명령의 내용은?" (X) → "웅진 계열사들이 받은 시정명령의 구체적인 범위와 위반 사실은 무엇인가요?" (O)

4. **소괄호 및 기계적 표현 엄격 제거**
   - `갑을관계(불공정거래)` → `갑을관계`
   - `공정거래(독과점/담합)` → `담합` 또는 `독과점` (내용에 따라 분리)
   - `소비자보호` → `소비자보호`
   - `(주)`, `주식회사` 등 회사명 및 기관명 접사 제거
    - **날짜/기간 표현 최적화 (Smart Display)**: 
        - 단일 날짜(예: 2020년 4월 8일)인 경우 "기간에 걸친" 대신 "당시의", "무렵 행한" 사용.
        - 범위형 기간인 경우, 앵커에서 `까지`, `동안` 등의 조사를 제거하고 "**2022년 7월부터 2024년 8월 기간에 걸친**"과 같이 템플릿과 자연스럽게 결합하여 중복 표현을 방지함.
    - **문맥 키워드 결합 (Context-Aware Anchor)**: 숫자(수치, 금액)만 앵커로 사용할 경우 문장이 기계적일 수 있으므로, 해당 숫자 주변의 도메인 키워드(예: 대출금리, 수수료, 점유율 등)를 함께 포착하여 "**대출금리 관련 7.89% 수준의...**"와 같이 자연스럽게 문맥을 형성함.

5. **자연스러운 질의 생성 (조사 및 단어 정제)**
   - `(주)`, `주식회사`, `(사)`, `사단법인`, `(재)`, `재단법인`, `(유)`, `유한회사` 등 불필요한 회사명 및 기관명 접사를 제거하고 자연스러운 호칭 사용.
   - **업종명 중복 표현 방지**: 업종명이 '기타 업종'과 같이 '업종'이라는 단어로 끝나는 경우, 질문 템플릿의 ' 업종인'과 결합 시 중복이 발생하므로 **'업종' 단어를 사전에 삭제**하여 자연스럽게 연결함. (예: "기타 업종 업종인" (X) → "기타 업종인" (O))
   - 단, '기타'와 같이 '업종'으로 끝나지 않는 일반 명사인 경우에는 삭제하지 않고 그대로 사용함.
   - '은/는/이/가/와/과' 등 조사를 문맥에 맞게 정확히 사용.

6. **사용자 시나리오 기반 비교/대조**
   - "두 의결서를 대조하시오"와 같은 명령형 대신, "A사와 B사의 위법 행위에는 어떤 차이가 있나요?"와 같은 탐색형 질문 사용.

5. **분류 체계의 구체화**
   - "공정거래(독과점/담합)" 등의 딱딱한 분류 대신 "담합", "시장지배력 남용", "하도급 갑질" 등 통용되는 용어 사용.

## 3. 정답 chunk 선정 및 answer_reason 작성 원칙

### 선정 기준 (우선순위)

1. **위법성/처분/주문 청크 중심**: 가급적 위법성 판단 근거와 구체적인 행위 사실을 담은 `위법성 판단` 관련 text 청크를 우선 **3개** 추출.
2. 추가로 `주문` 및 `처분` 섹션 청크를 각각 **1개씩** 더해 **총 5개** 할당 (Recall@5 변별력 확보 목적).
3. 복수 의결서의 경우 두 의결서에서 각각 **2~3개씩** 추출하여 총 **5~6개**의 정답 청크 구성.

### answer_reason 작성 기준
단순히 의결서 제목을 반복하는 것이 아니라, `{기업명}의 {위반분야}에 대한 위반 사실 및 법리 판단 근거`와 같은 일관성 있는 형태로 작성하여, 각 청크가 왜 질문의 정답이 되는지 명확히 설명할 것. (상위 분류 카테고리: 갑을관계, 공정거래, 소비자보호 등에 맞춰 정리)

## 4. 의결서 구조 패턴 (Hybrid JSON 공통)

```
주문          → CH-001 (대부분 1개, 시정명령/과징금 요약)
이유
├─ 1. 기초사실       → 피심인 현황, 시장 구조, 거래 관계
├─ 2. 사실의 인정 및 위법성 판단  → 행위사실, 법리 판단 (가장 많은 청크)
├─ 3. 처분           → 시정조치, 과징금 산정 기준/금액
└─ 4. 결론           → 최종 의결 (대부분 1~3개)
```

- **chunk_type**: `text` 또는 `table`
- **chunk_id 형식**: `DOC-{uuid}-CH-{3자리 순번}`

## 6. v4 개선 사항: 청크 원문 기반 쿼리 생성 (generation_260505.py)

### 6.1 v3의 문제점

v3의 쿼리는 메타데이터(기업명, 위반유형)만을 조합하여 생성된 범용 쿼리이다.

예시: `"○○가 담합을 한 구체적인 행위 사실과 그에 따른 처분 결과는 무엇인가?"`

이 쿼리는 동일 문서 내에서 담합 사실을 언급하는 모든 청크와 유사도가 높기 때문에, 선정된 정답 5개 청크 외의 청크도 정답으로 검색될 수 있다. 이 경우 Retrieval 평가 데이터셋으로서의 신뢰도가 낮아진다.

### 6.2 v4 핵심 변경: 앵커(Anchor) 추출 및 쿼리 삽입

v4는 정답 청크를 먼저 선정한 뒤, 해당 청크의 `page_content`에서 고유한 사실 정보(앵커)를 추출하여 쿼리에 직접 삽입한다.

**앵커 추출 우선순위**

| 순위 | 앵커 유형 | 예시 |
|---|---|---|
| 1 | 금액 | `3억 2천만원`, `500만원` |
| 2 | 기간/날짜 | `2018년 3월부터 2020년 1월까지` |
| 3 | 법조항 | `제19조 제1항` |
| 4 | 비율/배수 | `30%`, `1.5배` |
| 5 | 첫 의미 문장 (폴백) | `피심인은 2015년 8월...` |

**앵커 삽입 쿼리 예시**

```
v3 (범용): "○○가 담합을 한 구체적인 행위 사실과 그에 따른 처분 결과는 무엇인가?"

v4 (앵커): "○○가 2018년 3월부터 2020년 1월까지 기간 동안 행한 담합의
            구체적인 위반 행위 사실과 처분 내용은 무엇인가?"
```

앵커가 삽입된 v4 쿼리는 해당 기간/금액/법조항 정보를 담은 청크에만 높은 유사도를 가지므로, 정답 청크가 명확히 구분된다.

### 6.3 단일 문서 쿼리 생성 흐름

```
정답 청크 선정 (j_chunks, w_chunks, c_chunks)
        ↓
extract_anchor(primary_chunks)
  → 금액/기간/법조항/비율 순으로 탐색
  → 없으면 첫 의미 문장 추출
        ↓
build_single_query(anchor, anchor_type, ...)
  → 앵커 유형에 맞는 쿼리 템플릿 선택
  → 앵커를 쿼리 문장 내 삽입
        ↓
최종 쿼리: 정답 5개(최대 10개) 청크에만 연결된 구체적 질의
```

### 6.4 복수 문서 쿼리 생성 흐름

두 의결서의 정답 청크에서 각각 독립적으로 앵커를 추출하여 비교 쿼리를 생성한다.

```
w1 청크 → anchor1 추출
w2 청크 → anchor2 추출
        ↓
build_multi_query(anchor1, anchor2, ...)
  → 두 앵커를 모두 포함하는 비교 쿼리 생성
  → 앵커가 없는 경우 "위법성 판단 섹션"을 명시하여 범위 제한
```

### 6.5 출력 파일

| 파일 | 설명 |
|---|---|
| `eval_dataset_260505.json` | Recall@5(앵커 기반 검색) 및 BERTScore/F1 평가용 통합 데이터셋 |

> [!IMPORTANT]
> 본 파일 하나로 Retrieval 성능과 Generation 성능을 모두 평가할 수 있습니다. 
> - **Retrieval 평가**: `answer_chunks` 필드에 명시된 5개(복수 문서 5~6개) 청크 ID를 정답 집합으로 간주하여 Recall@5를 계산합니다.
> - **Generation 평가**: `reference_answer` 필드의 LLM 요약문을 골드 스탠다드로 활용하여 BERTScore, F1 등을 계산합니다.

## 7. Generation 평가 고도화: LLM 기반 요약 정답 생성

5개의 정답 청크(reference_context) 전체를 단순히 결합하여 평가 채점 기준(reference_answer)으로 사용하는 것이 아니라, 모델의 요약 능력과 정확도를 보다 정밀하게 측정하기 위해, 본 파이프라인은 LLM 기반 요약 정답 생성을 통해 Generation 평가를 고도화합니다.

### 7.1 전략 A와 전략 B 비교
- **전략 A (원문 통째 사용)**: `reference_answer` = `reference_context`. 구현이 단순하지만 사람이 읽기 불편하고 문맥이 너무 길어 평가 기준으로서 다소 부정확할 수 있습니다.
- **전략 B (LLM 기반 요약 정답)**: `reference_answer` = LLM이 작성한 3~5문장 내외의 요약된 모범 답안. 문맥(reference_context)과 질문(query)을 기반으로 LLM이 핵심 정답만 도출하여 골드 스탠다드로 활용합니다.

### 7.2 정답 생성 프롬프트 구성 (전략 B)
`generation_260505.py`에서 각 QA 데이터마다 Gemini API(gemini-3.1-flash-preview)를 호출하여 다음과 같은 지시로 정답을 생성합니다:

```text
[System]
당신은 공정거래위원회 심사관입니다.
주어진 의결서 문맥(Context)을 바탕으로 사용자의 질문(Query)에 대한 명확하고 간결한 정답을 작성하세요.
답변은 3~5문장 내외로 핵심만 요약해야 합니다. 반드시 Context 내용에만 기반해야 합니다.

[User]
Query: {질문}
Context: {5개 정답 청크 원문}
```

이 과정을 통해 생성된 요약 답변은 `eval_dataset_260505.json`의 `reference_answer` 필드에 저장되며, 추후 Ragas, TruLens 혹은 커스텀 평가 파이프라인에서 RAG 모델의 답변 품질(BERTScore, F1)을 채점하는 강력한 기준으로 작용합니다.

> [!NOTE]
## 8. 중복 청크 처리 원칙 (Cross-Document Duplicates)

동일한 사업자나 집단이 저지른 유사한 위반 행위가 여러 의결서(예: 교육청 담합 건 등)에 중복되어 기재된 경우, Retrieval 평가의 공정성을 위해 다음과 같이 처리함:

1. **동일 문서 내 중복**: 동일 의결서 내의 중복 내용은 그대로 유지함.
2. **타 문서 간 중복 (Ground Truth 확장)**:
   - 특정 위반 사실에 대해 질문했을 때, 내용이 완전히 동일한 청크가 다른 의결서에도 존재한다면, 해당 청크 ID들을 모두 `answer_chunks`에 포함시킴.
   - 이는 검색 모델이 어떤 문서를 먼저 찾아내더라도 정답으로 인정하기 위함임 (Recall 성능 보정).
   - `answer_reason`에는 "동일한 위반 사실이 명시된 타 의결서({의결서명})의 중복 청크"임을 명시함.

## 9. 생성 결과 요약 (v4 최종)
- **질의의 구체성**: 특정 시점, 수치, 법조항 등 '앵커'를 활용하여 수만 개의 청크 중 정답을 정확히 찾아낼 수 있는 변별력을 확보함.
- **자연어 최적화**: 회사명 접사 제거, 조사 정제, 문맥 키워드 결합을 통해 사람이 직접 작성한 듯한 전문적인 질의를 생성함.
